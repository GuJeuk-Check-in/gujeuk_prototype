# Gujeuk 첫 방문 수기 등록부 프로토타입

태블릿 PC 화면에 한 사람용 수기 등록지를 띄우고, 터치펜으로 큰 칸 안에 직접 작성하는 정적 프로토타입입니다.

## 실행

별도 설치 없이 `index.html`을 브라우저로 열면 됩니다.

```bash
open index.html
```

로컬 서버로 보고 싶다면 아래처럼 실행할 수 있습니다.

```bash
python3 -m http.server 5173
```

그다음 `http://localhost:5173`으로 접속합니다.

## 프로토타입 범위

- 한 사람만 작성할 수 있도록 방문 날짜/이름/나이/연락처/방문목적/개인정보 동의 여부/남자 동행인 수/여자 동행인 수 칸을 크게 표시합니다.
- 방문 날짜는 자동으로 채워집니다.
- 이름, 나이, 연락처, 동행인 수는 터치펜, 손가락, 마우스로 직접 필기합니다.
- 방문목적은 `GET https://api.taisu.site/purpose/all`에서 받은 목록 중 하나를 선택합니다.
- 방문목적 API에 연결할 수 없으면 IndexedDB에 마지막으로 저장된 목록이나 이전 등록에서 실제 사용했던 목적만 표시합니다.
- 인식 결과 확인 버튼을 누르면 필기 원본과 펜의 획 좌표를 캡처하고 실제 필기 인식을 실행합니다.
- 필기 인식 서버가 응답하지 않으면 `인식 없이 계속`을 눌러 빈 인식값과 원본 필기 전체를 우선 저장할 수 있습니다.
- 인식되지 않은 데이터는 `awaiting-recognition` 상태로 남고 연결 복구 후 재인식과 서버 전송을 시도합니다.
- 최종 저장 시 서버 전송보다 먼저 전체 데이터와 필기 원본을 브라우저 IndexedDB에 기록합니다.
- 브라우저에 영구 저장소 권한을 요청해 운영체제의 자동 저장공간 정리로부터 데이터를 최대한 보호합니다.
- 서버 전송에 실패한 데이터는 IndexedDB에 남겨두고 온라인 복귀, 30초 주기, 다시 연결 버튼을 통해 자동 재전송합니다.
- 서버 저장에 성공한 데이터도 `synced` 상태와 응답을 남겨 동기화 이력을 확인할 수 있습니다.
- 모든 등록 요청에는 중복 저장 방지를 위한 `Idempotency-Key`와 `X-Client-Request-Id` 헤더를 전달합니다.
- 서비스 워커가 화면 파일을 캐시하므로 웹 서버와 인터넷이 모두 끊겨도 한 번 실행했던 태블릿에서는 체크인 화면을 다시 열 수 있습니다.
- 이전 방문목적 목록도 전혀 없는 최초 실행 장치에서는 체크인을 중단하지 않고 `방문목적 목록 복구 후 확인 필요` 상태로 원본 데이터를 보관합니다.

## 필기 인식과 서버 연동 방향

- 현재 프로토타입은 Google Input Tools 필기 인식 서버에 획 좌표를 전송해 한글과 숫자 인식을 시도합니다.
- 인터넷 연결이 있으면 인식을 시도하고, 연결이 없거나 정확하지 않을 때는 직접 수정하거나 원본 필기 상태로 저장할 수 있습니다.
- 운영 서비스에서는 개인정보 처리 정책에 맞는 정식 필기 인식 API와 서버 프록시 구성이 필요합니다.
- 등록 데이터는 `POST https://api.taisu.site/user/sign-up`으로 전송을 시도합니다.
- 현재 수기 화면의 필드와 기존 서버 `SignupRequest` 필드가 완전히 같지 않으므로 운영 연결 전 `buildServerPayload()`의 최종 계약을 서버와 맞춰야 합니다.
- 완전한 중복 방지를 위해 서버에서도 `Idempotency-Key`를 저장하고 같은 키의 요청을 한 번만 처리해야 합니다.

실제 서버 연동 전 태블릿 수기 입력 흐름을 확인하기 위한 화면입니다.
